# Spring Miner 작업 규칙

## 1. 목적

이 문서는 `Spring Miner` 프로젝트에서 여러 명이 동시에 작업할 때 기준을 통일하고, 문서와 구현의 흐름을 빠르게 유지하기 위한 운영 규칙이다.

## 2. 회의 주기

- `주간 정례`: 매주 1회, 진행 상황 점검과 다음 주 우선순위 확정
- `데일리 체크`: 필요 시 비동기 메시지로 진행 상황 공유
- `의사결정 회의`: 범위 변경, 기술 선택, 일정 변경처럼 영향도가 큰 항목만 별도 논의
- `리뷰 회의`: 배포 또는 주요 마일스톤 전에 변경 사항과 리스크를 검토

## 3. 이슈 템플릿

모든 작업 이슈는 아래 항목을 포함한다.

- 제목: 한 줄 요약
- 배경: 왜 필요한지
- 목표: 이번 이슈의 성공 기준
- 범위: 포함/제외 항목
- 산출물: 문서, 화면, 코드, 테스트 결과 등
- 완료 기준: 끝났다고 판단하는 조건
- 리스크: 예상되는 제약과 대응
- 담당: 주 담당자와 협업자
- 기한: 목표 완료일

### 이슈 작성 예시

```text
제목: 명명 정보 추출기 결과 화면 개선
배경: 추출 결과를 더 빠르게 찾기 위해 검색 필터가 필요함
목표: 클래스/매퍼/테이블 결과를 유형별로 빠르게 탐색 가능하게 함
범위: 검색창, 필터, 결과 정렬
제외: 백엔드 저장, 권한 관리
산출물: HTML, JS, 문서
완료 기준: 샘플 데이터에서 3초 이내 탐색 가능
리스크: 정규식 정확도 부족
담당: A, B
기한: 2026-04-01
```

## 4. 작업 분해 기준

- 작업은 `문서`, `UI`, `분석 로직`, `검증`, `정리`로 먼저 나눈다.
- 하나의 작업은 가능한 한 `1일 이내` 또는 `반나절 단위`로 쪼갠다.
- 서로 다른 파일을 건드리는 작업은 의존 관계를 먼저 적는다.
- 추측이 필요한 작업은 먼저 가정과 한계를 문서에 남긴다.
- 구현 전에 입력, 출력, 예외, 완료 기준을 먼저 고정한다.

### 권장 분해 방식

1. 요구사항 정리
2. 화면/UX 초안
3. 데이터 구조 정의
4. 핵심 로직 구현
5. 검증 및 보정
6. 문서 반영

## 5. 브랜치 규칙

`git`이 초기화되지 않았더라도 아래 규칙을 기준으로 이름을 통일한다.

- 기능 브랜치: `feature/<ticket>-<short-name>`
- 수정 브랜치: `fix/<ticket>-<short-name>`
- 문서 브랜치: `docs/<short-name>`
- 실험 브랜치: `spike/<short-name>`

### 예시

- `feature/SM-101-naming-extractor-ui`
- `fix/SM-102-regex-column-detection`
- `docs/SM-103-working-rules`

## 6. 커밋 규칙

커밋 메시지는 짧고 행동 중심으로 작성한다.

- 형식: `<type>: <summary>`
- type 예시: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`
- summary는 현재형 동사 또는 명사구로 쓴다
- 하나의 커밋은 하나의 의미 있는 변경만 담는다

### 예시

- `feat: add naming extraction result cards`
- `docs: define working rules for parallel execution`
- `fix: handle empty mapper namespace values`

## 7. 리뷰 체크리스트

리뷰할 때는 아래 항목을 우선 확인한다.

- 요구사항과 구현이 일치하는가
- 범위를 벗어난 변경이 없는가
- 기존 문서와 설명이 충돌하지 않는가
- 예외 처리와 빈 상태가 있는가
- 정규식이나 파서의 한계가 명시되어 있는가
- 중복 코드나 불필요한 복잡도가 없는가
- 사용자 흐름이 끊기지 않는가
- 파일명, 함수명, 문서명이 규칙에 맞는가

## 8. 리스크 로그

모든 주요 리스크는 별도 로그로 관리한다. 로그는 다음 형식을 사용한다.

```text
ID: R-001
제목: 정규식 기반 추출의 정확도 부족
영향도: 높음 / 중간 / 낮음
상태: Open / Monitoring / Mitigated / Closed
설명: 복잡한 Java 문법과 동적 SQL에서 누락 가능성이 있음
대응: AST 도입 검토, 샘플 케이스 추가, 예외 목록 관리
담당: A
갱신일: 2026-03-25
```

### 운영 원칙

- 리스크는 발견 즉시 기록한다.
- 상태가 바뀌면 반드시 갱신한다.
- 대응이 끝난 항목은 원인과 결과를 한 줄로 남긴다.

## 9. 문서 명명 규칙

문서는 UTF-8 기준으로 저장하고, 아래 규칙을 따른다.

- 문서 파일명은 `kebab-case.md`를 기본으로 한다.
- 목적이 분명한 이름을 사용한다.
- 날짜가 필요하면 `YYYY-MM-DD` 형식을 사용한다.
- 중복 가능성이 있으면 접두어를 붙인다.

### 예시

- `planning.md`
- `requirements.md`
- `mvp.md`
- `ui-wireframe.md`
- `working-rules.md`
- `2026-03-25-decision-log.md`

## 10. 다중 작업 원칙

- 여러 사람이 동시에 작업할 때는 파일 소유권을 먼저 나눈다.
- 같은 파일을 동시에 수정해야 하면 수정 범위와 순서를 먼저 합의한다.
- 문서 변경은 구현 변경과 분리해서 리뷰한다.
- 새 문서를 만들 때는 기존 문서와 중복되지 않도록 범위를 먼저 확인한다.

## 11. 종료 기준

작업 종료 시 아래 항목이 충족되어야 한다.

- 결과 파일이 저장되었다.
- 변경 요약이 남았다.
- 리스크가 있으면 로그에 기록되었다.
- 다음 사람이 이어받을 수 있을 정도로 문서가 정리되었다.
