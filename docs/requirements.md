# Spring Miner 요구사항 정의서

## 1. 프로젝트 목표

기존 `Spring Framework` 기반 시스템의 소스 및 DB 관련 파일에서 주요 명명 정보를 자동 추출하고, 이를 검색 및 조회할 수 있는 내부용 웹 도구를 제공한다.

## 2. 대상 사용자

- 개발자
- 유지보수 담당자
- 신규 투입 인력
- 필요 시 기획 및 운영 담당자

## 3. 문제 정의

기존 레거시 시스템에서는 다음과 같은 문제가 발생한다.

- 클래스, 메소드, 매퍼, 테이블 구조를 빠르게 파악하기 어렵다.
- 코드와 DB 간 연결 지점을 찾는 데 시간이 많이 든다.
- 명명 규칙이 일관되지 않은 경우 전체 구조 이해가 더 어려워진다.
- 신규 인력이 시스템 구조를 학습하는 데 시간이 오래 걸린다.

## 4. 기능 요구사항

### FR-01. 분석 대상 입력

- 사용자는 프로젝트 파일을 업로드할 수 있어야 한다.
- 사용자는 특정 소스 경로를 지정하여 분석할 수 있어야 한다.
- 초기 버전에서는 파일 업로드 방식만 우선 적용할 수 있다.

### FR-02. 분석 대상 파일 유형

- 시스템은 `Java`, `XML`, `SQL`, `TXT` 파일을 분석할 수 있어야 한다.
- 향후 `JSP`, `properties`, `yml` 확장이 가능해야 한다.

### FR-03. 명명 정보 추출

- 시스템은 다음 정보를 추출할 수 있어야 한다.
- 클래스명
- 인터페이스명
- 메소드명
- 변수명
- 매퍼 namespace
- mapper statement id
- 테이블명
- 컬럼명

### FR-04. 결과 조회

- 추출 결과는 유형별로 구분되어 보여야 한다.
- 결과는 중복 제거되어야 한다.
- 결과는 검색 가능해야 한다.
- 결과는 정렬 가능해야 한다.

### FR-05. 상세 정보 조회

- 사용자는 특정 항목을 선택했을 때 원본 파일명과 경로를 확인할 수 있어야 한다.
- 가능하다면 라인 번호 또는 코드 일부를 함께 확인할 수 있어야 한다.

### FR-06. 결과 활용

- 사용자는 결과를 복사할 수 있어야 한다.
- 사용자는 CSV 또는 Excel 형태로 결과를 내려받을 수 있어야 한다.

### FR-07. 재분석

- 사용자는 동일 대상에 대해 반복 분석을 수행할 수 있어야 한다.
- 추후 버전에서는 분석 결과 비교가 가능해야 한다.

## 5. 비기능 요구사항

### NFR-01. 사용성

- 내부 사용자가 별도 교육 없이 사용할 수 있어야 한다.
- 결과 화면은 유형별 구분이 명확해야 한다.

### NFR-02. 성능

- 소규모 프로젝트는 수 초 내 분석 결과를 확인할 수 있어야 한다.
- 대규모 프로젝트는 단계적 로딩 또는 비동기 처리 구조를 고려해야 한다.

### NFR-03. 확장성

- 추출 대상 유형을 추가하기 쉬운 구조여야 한다.
- 향후 백엔드 연동 및 저장 기능을 추가할 수 있어야 한다.

### NFR-04. 정확성

- 초기 버전은 정규식 기반으로 시작할 수 있으나, 복잡한 Java 문법 및 동적 SQL 대응 한계를 명시해야 한다.
- 고도화 단계에서는 AST 또는 파서 기반 분석 도입을 검토해야 한다.

## 6. MVP 범위

### 포함

- 파일 업로드 기반 분석
- Java / XML / SQL 텍스트 분석
- 클래스명, 메소드명, 변수명, namespace, mapper id, 테이블명, 컬럼명 추출
- 웹 UI에서 목록 조회
- 검색 및 필터

### 제외

- 프로젝트 디렉터리 자동 스캔
- 라인 번호 정밀 추적
- 관계도 시각화
- 저장형 이력 관리
- 권한 관리

## 7. 향후 확장 요구사항

- 프로젝트 경로 직접 스캔
- 배치 분석
- 히스토리 비교
- 호출 관계 및 의존성 시각화
- 명명 규칙 점검
- 관리자용 정책 설정

## 8. 성공 기준

- 사용자가 레거시 프로젝트의 주요 클래스, 매퍼, 테이블, 컬럼 정보를 빠르게 검색할 수 있다.
- 기존 수작업 대비 구조 파악 시간을 줄일 수 있다.
- 최소 기능 제품으로 내부 사용 가능 수준의 탐색 경험을 제공한다.
