# Spring Miner Agent Guide

## 프로젝트 개요

Spring Miner는 기존 `Spring Framework` 기반 코드와 DB 자산에서 변수명, 클래스명, 메소드명, 매퍼명, 네임스페이스, 테이블명, 컬럼명을 추출해 내부에서 조회하고 분석하는 인하우스 도구다.

이 저장소는 문서화, 정적 UI, 추출 로직, 이후 확장 가능한 분석 파이프라인을 함께 다룬다. 작업자는 혼자가 아니며, 변경은 항상 다른 에이전트의 작업과 충돌 가능성을 전제로 한다.

## 에이전트 역할

- `planner`: 요구사항, 화면 흐름, MVP 범위를 정의하고 문서 구조를 유지한다.
- `ui-agent`: HTML, CSS, 프론트엔드 상호작용을 담당한다.
- `extract-agent`: Java, XML, SQL에서 명명 정보를 추출하는 로직을 담당한다.
- `docs-agent`: `README`, 기획서, 요구사항, 운영 규칙 문서를 관리한다.
- `review-agent`: 변경 사항의 결함, 누락, 회귀 위험을 점검한다.

## 병렬 작업 규칙

- 서로 다른 파일과 독립된 범위는 병렬로 진행한다.
- 같은 파일을 동시에 수정하지 않는다.
- 같은 기능을 건드리더라도 책임 범위를 먼저 나눈다.
- 병렬 작업 시작 전에는 각 에이전트의 파일 소유 범위를 명확히 적는다.
- 충돌 가능성이 있으면 병렬보다 순차 작업을 우선한다.

## 소유권 규칙

- 한 작업 단위는 하나의 주 책임 파일 또는 기능 범위만 가진다.
- 문서 작업은 `docs/` 아래에서 우선 처리한다.
- UI 변경은 `index.html`, `styles.css`, `app.js`를 하나의 묶음으로 보되, 수정 전 범위를 합의한다.
- 로직 변경과 문서 변경을 섞지 않는다.
- 다른 에이전트의 미완료 변경을 되돌리지 않는다.

## 조정 프로토콜

1. 작업 시작 전에 담당 범위와 예상 결과물을 짧게 선언한다.
2. 변경이 필요한 파일명을 먼저 공유한다.
3. 공통 인터페이스, 데이터 구조, 용어는 문서에서 먼저 고정한다.
4. 구현 중 충돌이 생기면 마지막 편집자가 아니라 범위 책임 기준으로 조정한다.
5. 완료 후에는 변경 파일, 검증 방법, 남은 위험을 짧게 보고한다.

## 완료 기준

- 요구한 산출물이 실제 파일로 저장되어 있다.
- 문서와 구현의 용어가 일치한다.
- 최소한의 검증을 통과한다.
- 후속 작업자가 이어서 작업할 수 있을 만큼 맥락이 정리되어 있다.
- 불확실한 부분은 추측으로 덮지 않고 문서에 남긴다.

## 안전 편집 규칙

- `apply_patch`로만 수동 편집한다.
- 불필요한 대규모 재작성은 피한다.
- 기존 사용자 변경은 유지한다.
- 되돌리기나 강제 초기화는 하지 않는다.
- 한 번에 여러 파일을 바꾸더라도 목적이 하나여야 한다.
- 새 파일을 만들 때는 ASCII를 기본으로 한다.

## 권장 작업 분해

- 1단계: 기획 문서와 요구사항을 고정한다.
- 2단계: 화면 구조와 데이터 항목을 정리한다.
- 3단계: 추출 대상과 파서 규칙을 정의한다.
- 4단계: UI와 추출 로직을 분리해 구현한다.
- 5단계: 샘플 데이터와 검증 시나리오를 추가한다.
- 6단계: 결과 다운로드, 상세 보기, 이력 기능을 확장한다.

## 킥오프 체크리스트

- 작업 범위가 문서로 존재하는가.
- 이번 작업이 문서인지 구현인지 구분했는가.
- 수정 대상 파일을 명시했는가.
- 다른 에이전트와 겹치는 파일이 없는가.
- 완료 기준과 검증 방법을 정했는가.
- 실패 시 되돌릴 것이 아니라 보강할 항목을 정했는가.

## 운영 원칙

- 짧고 명확하게 보고한다.
- 추측 대신 근거를 남긴다.
- 문서와 구현을 함께 관리한다.
- 협업 속도보다 충돌 방지와 재작업 최소화를 우선한다.
